40  * </p

42  * <b>Glossary of terms</b>
43  *
44  * </p <ul> <li>
45  *
46  * <b>Server</b>: A configuration management server that can be highly-available. The configuration management server runs
47  * on an Amazon Elastic Compute Cloud (EC2) instance, and may use various other AWS services, such as Amazon Relational
48  * Database Service (RDS) and Elastic Load Balancing. A server is a generic abstraction over the configuration manager that
49  * you want to use, much like Amazon RDS. In AWS OpsWorks CM, you do not start or stop servers. After you create servers,
50  * they continue to run until they are
51  *
52  * deleted> </li> <li>
53  *
54  * <b>Engine</b>: The engine is the specific configuration manager that you want to use. Valid values in this release
55  * include <code>Chef</code> and
56  *
57  * <code>Puppet</code>> </li> <li>
58  *
59  * <b>Backup</b>: This is an application-level backup of the data that the configuration manager stores. AWS OpsWorks CM
60  * creates an S3 bucket for backups when you launch the first server. A backup maintains a snapshot of a server's
61  * configuration-related attributes at the time the backup
62  *
63  * starts> </li> <li>
64  *
65  * <b>Events</b>: Events are always related to a server. Events are written during server creation, when health checks run,
66  * when backups are created, when system maintenance is performed, etc. When you delete a server, the server's events are
67  * also
68  *
69  * deleted> </li> <li>
70  *
71  * <b>Account attributes</b>: Every account has attributes that are assigned in the AWS OpsWorks CM database. These
72  * attributes store information about configuration limits (servers, backups, etc.) and your customer account.
73  *
74  * </p </li> </ul>

77  *
78  * </p
